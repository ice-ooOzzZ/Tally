# 智能选股分析系统 PRD（v0.9 初稿）

## 1. 项目概述

### 1.1 背景与目标

个人投资者面对 A股 5000+ 只、美股数千只股票，无法人工持续跟踪。本系统将选股方法论固化为程序：**每周**按基本面标准维护受监控的股票池，**每日**对池内股票执行规则化分析，输出"建议买入/建议卖出/观望"及**可解释的触发原因**，并通过看板和 IM 推送触达用户。

核心设计原则：

1. **可解释**：每个信号必须附带人能读懂的原因，拒绝黑盒
2. **可验证**：任何策略上线前必须通过历史回测；上线后持续追踪建议的真实表现
3. **可扩展**：规则即插件，新增一条选股规则不改动引擎代码
4. **自用优先，预留产品化**：单机可跑，但模块边界与接口按未来服务化标准设计

### 1.2 非目标（一期明确不做）

- 不做自动下单/实盘交易接口（合规与风险原因，仅输出建议）
- 不做分钟级/tick 级高频数据与日内策略
- 不做多用户账号体系（但架构预留）
- 不做 ML 预测模型（先验证规则策略基线）

### 1.3 目标用户与成功指标

一期用户为开发者本人。成功指标：

- 系统层面：每日报告按时产出率 > 99%；单日全流程运行时间 < 15 分钟
- 策略层面：回测（近5年）年化收益跑赢基准指数（沪深300 / 标普500），最大回撤 < 基准 1.2 倍，信号胜率 > 45% 且盈亏比 > 1.5
- 使用层面：每个信号的原因可读、可复核，用户可在 3 分钟内看完日报

## 2. 总体架构

### 2.1 对原始四层划分的审核

用户原始划分：股票层 / 策略层 / 数据层 / 运行层。审核意见：

| # | 意见 | 调整 |
|---|---|---|
| 1 | 数据层是被股票层和策略层共同消费的基础设施，应显式置底 | 分层顺序调整为自底向上 |
| 2 | "运行层"混合了调度执行与展示通知两种职责，变更频率和技术栈完全不同 | 拆分为调度层与展示层 |
| 3 | 缺回测验证：不回测，规则的分值与阈值全是主观臆断 | 新增回测与验证层（一期必做） |
| 4 | 缺信号追踪：系统上线后无法回答"它到底准不准" | 归入验证层，记录每次建议的后续表现 |
| 5 | 缺风控辅助：只说买不说止损价和仓位的系统是残缺的 | 归入策略层输出 |

### 2.2 调整后的六层架构

```
┌─────────────────────────────────────────────────┐
│  L6 展示与通知层   Web看板(Streamlit) / Telegram Bot │
├─────────────────────────────────────────────────┤
│  L5 调度层         APScheduler 任务编排/重试/告警    │
├─────────────────────────────────────────────────┤
│  L4 回测与验证层   历史回测引擎 / 信号追踪成绩单       │
├─────────────────────────────────────────────────┤
│  L3 策略层         规则引擎 / 打分合成 / 风控辅助     │
├─────────────────────────────────────────────────┤
│  L2 股票池层       全市场初筛 / 入池出池 / 白黑名单    │
├─────────────────────────────────────────────────┤
│  L1 数据层         数据源适配 / 同步管道 / SQLite存储 │
└─────────────────────────────────────────────────┘
横切关注点：配置中心(YAML+pydantic) / 结构化日志 / 异常告警
```

数据流：`L1 数据更新 → L2 周度筛池 → L3 日度信号 → L4 记录追踪 → L6 报告推送`，全部由 L5 编排。

## 3. 分层详细设计

### 3.1 L1 数据层

**数据源**（一期全部免费源，适配器模式封装，可替换）：

| 市场 | 数据 | 源 | 频率 |
|---|---|---|---|
| A股 | 全市场实时快照（价格/PE/PB/市值/成交额） | AkShare（东财） | 每周筛池时 |
| A股 | 日K（前复权） | AkShare | 每日收盘后 |
| A股 | 财务指标（ROE/营收/现金流） | AkShare 财报接口 | 每季 |
| 美股 | 成分股清单（S&P500） | Wikipedia/静态维护 | 每月 |
| 美股 | 基本面（PE/ROE/市值） | yfinance | 每周 |
| 美股 | 日K | yfinance | 每日 |

**同步方式**：定时批量拉取（非实时订阅）。池内股票日K更新用线程池并发（受限频约束，并发度可配，默认 4，请求间隔可配）；全市场快照单次请求。所有拉取带指数退避重试（3 次）。**增量更新**：日K只拉本地缺失日期区间。

**存储**：一期 SQLite（零运维、单文件、pandas 友好），通过 Repository 层隔离 SQL，预留 PostgreSQL 迁移。核心表：

- `kline(code, market, date, open, high, low, close, volume, PRIMARY KEY(code,market,date))`
- `fundamentals(code, market, report_date, pe, pb, roe, market_cap, ...)`
- `pool_history(code, market, action[enter/exit], date, reason)`
- `signals(id, date, code, market, advice, score, reasons_json, price_at_signal)`
- `signal_tracking(signal_id, days_after, price, return_pct)`
- `backtest_runs(id, strategy_config_json, period, metrics_json, created_at)`

**数据质量**：停牌日无K线为正常缺失不补；退市股票标记冻结；复权统一前复权；财报数据校验非负约束；每次同步后跑数据健康检查（覆盖率、最新日期）并记录。

### 3.2 L2 股票池层

**双重筛选**：硬条件过滤 + 排序取前 N。

- A股硬条件（默认值，全部可配）：总市值 ≥ 100亿；0 < 动态PE ≤ 40；PB < 8；20日日均成交额 ≥ 1亿；上市满 1 年；非 ST
- 美股硬条件：市值 ≥ $10B；0 < PE ≤ 40；ROE ≥ 12%；日均量 ≥ 100万股
- 排序：按市值降序（一期），预留多因子综合排序接口
- 池大小：A股 50 + 美股 30（可配）

**出入池机制**：

- 周度运行；新满足者入池并记录入池理由
- 出池宽限：连续 2 周不满足才出（防临界抖动）；急剧恶化（如 PE 转负）立即出
- 手工白名单（永不出池，用户自选股）与黑名单（永不入池）
- 全部变动写入 `pool_history`，可回溯任意日期的池子快照（回测需要）

### 3.3 L3 策略层

**规则引擎**——核心扩展点，设计为插件式：

```python
class Rule(ABC):
    name: str          # 规则名
    category: str      # trend/momentum/volume/valuation/risk
    default_weight: float

    @abstractmethod
    def evaluate(self, ctx: StockContext) -> RuleResult:
        """ctx 提供该股的K线+指标+基本面+池信息
        返回 RuleResult(score: float, reason: str|None)"""
```

- 规则通过注册器自动发现（装饰器 `@register_rule`），新增规则=新增一个类文件，零改动引擎
- 权重与启停在 YAML 配置：`rules: {ma_cross: {enabled: true, weight: 1.0}, ...}`
- 一期内置规则集（沿用已验证原型逻辑）：均线金叉/死叉与多空排列、MA20 位置、MACD 交叉、RSI 超买/超卖回升、量价配合（3日方向）、10日急跌、250日价格分位

**信号合成**：加权总分，`score >= +3` 买入 / `<= -3` 卖出 / 其余观望（阈值可配）。报告展示每条命中规则的得分与原因。

**风控辅助输出**（随买入信号一起给出，不是强制指令）：

- 建议止损价：`min(入场价 × (1-8%), 20日低点, 入场价 - 2×ATR14)` 取最保守
- 建议仓位：基于 2% 单笔风险规则 —— `仓位 = 总资金×2% ÷ (入场价-止损价)`，输出为总资金百分比
- 持仓提醒：池内有历史买入信号的股票触发卖出信号时高亮提示

### 3.4 L4 回测与验证层（一期必做）

**回测引擎**：事件驱动、逐日回放，严防未来函数：

- 用 `pool_history` 重建历史每一天的股票池（避免存活者偏差的关键）
- T 日收盘数据出信号，T+1 日开盘价成交（A股考虑涨停无法买入/跌停无法卖出）
- 交易成本：A股双边佣金 0.05%（最低5元）+ 卖出印花税 0.05% + 滑点 0.1%；美股按 IBKR 费率
- 组合层面：等权持有全部买入信号，单票上限 10%，最多同时持有 15 只

**输出指标**：年化收益、最大回撤、夏普比率、胜率、盈亏比、换手率、对比基准（沪深300/标普500）超额收益、逐年分解。

**防过拟合纪律**：参数调优只允许在训练段（如 2019-2023）做，2024 至今为样本外验证段；单次实验全配置与结果入库（`backtest_runs`），保留完整实验记录。

**信号追踪（实盘影子）**：每日信号生成后，T+1/T+5/T+20 日自动记录该股后续涨跌，看板展示滚动胜率成绩单——回答"这系统最近到底准不准"。

### 3.5 L5 调度层

- 一期用 APScheduler 常驻进程（比 cron 好处：任务依赖、失败捕获、统一日志）
- 任务流水线：
  - 每交易日 16:30（A股收盘后）：数据更新 → 信号生成 → 信号追踪更新 → 报告生成 → 推送。美股因时差在北京时间次日 05:30 单独跑，日报合并展示
  - 每周一 08:00：全市场筛池
- 失败处理：单任务重试 3 次（指数退避）；最终失败发告警到 Telegram；某只股票数据失败不阻塞整体，报告中标注"N 只分析失败"
- 交易日历：A股用 AkShare 交易日历，美股用 pandas_market_calendars，非交易日跳过

### 3.6 L6 展示与通知层

**Web 看板**（一期 Streamlit，本地 `localhost:8501`，预留 FastAPI+前端演进）：

1. **今日信号页**：买入/卖出卡片（股票、得分、原因列表、止损价、建议仓位），观望表格
2. **股票池页**：当前池子、本周出入池变动及理由、按市场/行业分布
3. **个股详情页**：K线图（ECharts/plotly，叠加 MA20/60、买卖信号标记点）、历史信号列表、基本面摘要
4. **策略成绩单页**：信号追踪滚动胜率、近90日信号的 T+5/T+20 平均收益、与基准对比
5. **回测实验室页**：选时间段与参数 → 跑回测 → 展示资金曲线、回撤曲线、指标表、交易明细

**推送**（Telegram Bot 一期 / Slack Webhook 二期，二者接口同构）：

- 每日收盘后推摘要：`📈 今日信号：买入2 卖出1 | 池内74只 | 明细见看板`，买卖信号附一句话原因
- 告警推送：任务失败、数据源异常
- 推送为"摘要+引导到看板"，不在 IM 里塞全文

## 4. 技术选型

| 项 | 选型 | 理由 |
|---|---|---|
| 语言 | Python 3.11+ | 量化生态最全 |
| 数据 | AkShare + yfinance | 免费、社区活跃；适配器隔离便于换付费源 |
| 存储 | SQLite → 预留 PostgreSQL | 单机零运维；Repository 层隔离迁移成本 |
| 计算 | pandas + numpy | 标准选择 |
| 调度 | APScheduler | 进程内依赖编排，比 cron 可控 |
| 看板 | Streamlit → 预留 FastAPI+React | 一期开发速度优先 |
| 图表 | plotly | Streamlit 原生集成，K线交互好 |
| 配置 | YAML + pydantic | 类型安全，配置即文档 |
| 推送 | python-telegram-bot | 一期 |

**产品化预留**（一期不实现但不设障碍）：各层间通过明确的 Python 接口交互，禁止跨层直接查库；配置与代码分离；数据层/策略层无状态化，未来可拆服务；信号与回测结果带 schema 版本号。

```
├── strategy/
│   ├── engine.py           # 规则加载与打分合成
│   ├── rules/              # 每条规则一个文件（扩展点）
│   ├── indicators.py       # 技术指标计算
│   └── risk.py             # 止损价/仓位建议
├── backtest/
│   ├── engine.py           # 事件驱动回测
│   ├── broker.py           # 成交模拟/成本模型
│   └── metrics.py          # 绩效指标
├── tracking/               # 信号追踪
├── scheduler/              # APScheduler 任务定义
├── dashboard/              # Streamlit 页面
├── notify/                 # telegram.py / slack.py
└── tests/                  # 单元测试（规则、回测防未来函数为重点）
```

## 6. 里程碑

| 阶段 | 内容 | 工作量估计 | 验收标准 |
|---|---|---|---|
| M1 | 数据层：双市场同步管道 + SQLite + 健康检查 | 1.5 周 | 池内股票 3 年日K完整落库，每日增量更新稳定 |
| M2 | 股票池 + 策略层：筛池、规则引擎、信号与风控输出 | 1 周 | 每日产出带原因的信号，新增规则无需改引擎 |
| M3 | 回测层：引擎 + 指标 + 样本外验证 | 2 周 | 5 年回测跑通，防未来函数测试通过，产出策略基线报告 |
| M4 | 调度 + 看板 + 推送 | 1.5 周 | 全流程无人值守运行一周，看板五个页面可用 |
| M5 | 试运行与调优：信号追踪积累、规则权重按回测调整 | 持续 4 周 | 滚动胜率与回测预期偏差 < 10 个百分点 |

## 7. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 免费数据源限频/接口变更 | 同步失败 | 适配器隔离，重试+告警，必要时切 Tushare Pro（¥200/年）|
| 规则策略回测不赚钱 | 项目核心价值存疑 | M3 设为决策门：基线不达标则调整规则集而非硬上 |
| 过拟合（调参调到历史最优） | 实盘失效 | 样本外验证纪律 + 信号追踪持续对照 |
| 单机运行电脑关机漏跑 | 报告断档 | 补跑机制（启动时检查漏跑日期）；二期可迁云 |
| 合规 | 若产品化涉投顾牌照 | 一期自用规避；所有输出带免责声明；产品化前法务评估 |
