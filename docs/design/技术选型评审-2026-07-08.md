## 1. 一句话总纲

双市场引入的是「领域复杂度」(数据脆弱 + 中美时区两段操作 + 退市/复权),**不是「基础设施复杂度」**——逐项核对,没有任何一项新增引入新的常驻组件。六大主题选型答案一个都不改,反被双市场进一步印证;v2.2 的全部新工作量落在数据接入层与「隔离多市场差异的抽象 + 纪律」上,正解是 MarketProfile 抽象 + 硬约束护栏,而非加组件。弃坑风险确实抬高,但抬高在「数据运维 + 用户时间」两条线,不在「技术栈维护」。

## 3. 六大主题修订

### 主题一:前端框架
- **v2.0 延续:** 三页 Streamlit 单体、plotly、直读 Repository(查询加 `market` 入参)、回填四件套(`st.form`+`session_state`+`st.fragment`+提交写 upsert)、拆分临界点仍在二期。四方一致。
- **v2.2 修订:**
  1. **交互重心调整(产品):** 美股深夜 21:30 执行、双市场合计操作须 ≤10min,因此**明确 Slack 日报摘要为日常主界面、看板降为按需、回填优先 CLI**,以保住「回填率 ≥70%」成功标准。
  2. **实现级替换(前端裁决):页②/⑤ 市场切换用 `st.segmented_control` 替换裸 `st.tabs`。** 理由:规避 `st.tabs` 两个坑——Tab 内交互后弹回首 Tab、两市场全量渲染;`segmented_control` 天然懒加载只渲染选中市场。页⑤ 市场×策略**不用嵌套 Tab**,用 `segmented_control(市场)+ 组内分栏`。
  3. **widget key 全加 market 前缀**(如 `f"nav_range_{market}"`),防跨市场串状态。
  4. **`expired` 是第 4 展示态,非第 4 回填态:** 前端只读 `note=='expired'` 用 `disabled=True` 灰显,**过期判定留后端 runner,前端绝不自己算日历**。市场徽标/币种从 `MarketProfile.currency` 取,纯展示零成本。

### 主题二:后端语言
- **v2.0 延续:** Python 3.11+ 单语言,永不换语言。四方一致并强化。
- **v2.2 修订:** **版本纪律从「建议」升「必须」** —— `requires-python=">=3.11,<3.13"` 不变;新增 **yfinance 必须 pin 精确版本(`yfinance==0.2.xx` 而非 `>=`)** 作为脆弱依赖管理。

### 主题三:Web 框架 / API 层
- **v2.0 延续:不变,强化。** 不引 FastAPI;`tally run --market` 的参数化正好 1:1 映射未来 HTTP query。四方一致。
- **v2.2 修订:** 无。

### 主题四:数据库
- **v2.0 延续:** 裸 `sqlite3` + 手写 SQL,不引 ORM/SQLAlchemy;PRAGMA(WAL / busy_timeout=30000 / synchronous=NORMAL / foreign_keys=ON)不变;**schema 变更是纯 DDL 文本 + 允许删库重建,明确不需 Alembic**;PG 触发条件不变(美股全历史 ~123 万行 kline ≈120–150MB,总库悲观 ≤1.5GB,仍远在舒适区,数据量不构成新触发)。
- **v2.2 修订(2 条新硬约束,见 §5):** 跨进程写串行化(flock,T4.3)、market 维度收口(T4.3/T4.4)。此外:
  - **US 日K split/dividend 全量重拉必须原子:** 封成单 writer 队列任务、包在一个 `BEGIN...COMMIT`,Repository 加 `replace_kline_series` 原子方法(单写线程天然保证)。
  - 新增监控点:美股全量重拉的**写放大**、T3.6a 一次性历史大回补基线。
  - `kline.source` 血缘列;`yfinance info` 的 stale 标注缓存**落 SQLite 既有表,非新机制**。

### 主题五:缓存
- **v2.0 延续:** 三层缓存(`fundamentals_derived` 表 + `functools.lru_cache` + pandas 内存)+ `st.cache_resource`/`st.cache_data`,不引 Redis/diskcache。估值分位移二期后反而更简。
- **v2.2 修订(正确性要求):**
  1. **cache key 必须 `(market, trade_date)`** —— 两市场交易日历不同,「最新交易日」分市场不同,不带 market 会串数据。
  2. 两市场独立更新独立失效;**写后仍手动 `st.cache_data.clear()` 全清**。

### 主题六:中间件 / 调度 / 通知
- **v2.0 延续:** 无外部中间件;调度 **launchd**(推翻 APScheduler)结论不变。**两份 launchd 不是任务队列**,是两条独立、带 catch-up 游标的幂等 cron。
- **通知渠道(项目负责人 2026-07-08 决策,推翻上一轮 Slack 裁决):改用 Telegram。** 落地方式仍为 **`requests` POST 到 Telegram Bot API `sendMessage`,不引入 `python-telegram-bot` SDK**(与「Slack 用 requests 不引 slack_sdk」同构)。配置从单一 `SLACK_WEBHOOK` 改为 **`TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID`(均走 `.env`)**。⚠️ 此决策与 spec v2.2 冲突(spec 通篇为 Slack:§0 密钥、§10 `slack_webhook`、§8.3/8.4),需同步修订 spec。
- **v2.2 修订(令牌桶升级):**
  1. **令牌桶 2 → 4,升级为 `RateLimiterRegistry`(per-source 注册表)**:速率型(tushare/sec)用令牌桶,并发型(akshare/yfinance)用 `Semaphore(2)`。
  2. **yfinance 错峰分批(批间隔 5min)是令牌桶表达不了的有状态批调度器,须自研 ~50 行**(标准库 + tenacity)。
  3. 并发仍 `ThreadPoolExecutor` 不用 asyncio(yfinance/SEC/Stooq 全同步阻塞 HTTP);重试仍 tenacity。

## 4. v2.2 新增选型点决策

### A) MarketProfile + market_overrides 机制 —— 四方一致强烈背书
- **定位:** 隔离多市场差异的正确抽象,是铁律二「策略代码一份」的直接实现(组合优于继承)。对单人可维护性生死攸关。
- **架构师专项评价 + 3 点收敛(纳入约束):**
  1. **禁 `if market==` 做成 AST/grep 可测的不变量,是最亮一笔** → 列 **M3 硬验收门**(「零 `if market==` 违例」,broker/profile 豁免是正确收口)。
  2. **深合并 + null 删除是迷你配置 DSL**,须在 **T0.5.2 锁死列表字段「整体替换而非逐元素合并」语义**,并提供 `tally config dump --strategy s1 --market US` 打印合并终值。
  3. **能力开关(`has_price_limit` 等)不得增殖成 God-object**:新增字段须是「市场结构事实」而非「功能开关」;不为单一取值字段建模(YAGNI)。
- **实现方案(后端):** config 深合并 = **15 行纯函数 `deep_merge`,在 dict 阶段合并完再 `model_validate`**(不用 `model_copy` 浅合并);每(策略×市场)产出一份 frozen 参数;pydantic `extra=forbid` 捕 typo。`system/portfolio/pool/costs` 的 market 嵌套用 `dict[str, MarketConfig]` 建模。

### B) SEC EDGAR 客户端 —— 裁决:自研 requests(~50 行),拒绝库
- **主源裁决:** SEC XBRL 为美股基本面唯一主源(唯一满足全历史 + 真实 filed 日 + 含退市的免费源)。
- **自研 vs 库:**
  - **明确拒绝 `edgartools`**:功能过剩,其抽象与 spec 的 tag/生效日口径打架。
  - **`sec-edgar-api` 仅作退路**:其内建限流与全局令牌桶冲突。
  - **结论:自研 ~50 行 requests 客户端。**
- **落地要点:** ticker→CIK 补零 10 位;收入 tag 三级 fallback 落 config(`Revenues`→`RevenueFromContractWithCustomerExcludingAssessedTax`→`SalesRevenueNet`);**生效日 = filed + 1 交易日**(`pandas_market_calendars` 算);Q4 单季派生(10-K 年度 − 前三季);**User-Agent 缺失启动即报错**。

### C) 多令牌桶注册表 —— 升级为 `RateLimiterRegistry`
速率型令牌桶(tushare 400/min、sec ≤8rps)+ 并发型 Semaphore(akshare/yfinance concurrency=2)混合;yfinance 有状态错峰批调度器(100 只/批、批内并发 2、间隔≥0.6s、批间隔 5min、429 退避)自研 ~50 行。

### D) config 深合并 DSL
见 §4-A 实现方案(`deep_merge` 15 行 + `tally config dump` 工具,T0.5.2)。

### E) 交易日历双后端分派
`common/calendar.py` 从单实现变双后端,经 **`calendar(market)` 分派**:CN=Tushare trade_cal 落库查询 / US=`pandas_market_calendars`(NYSE,含半日市)。

### F) 辅助数据源 —— 全部薄封装,不引重库
- **`lxml`**:仅为 `pd.read_html` 解析 Wikipedia 单页(S&P500 成分 + Selected changes)。
- **`fja05680/sp500`**:用 requests 拉 CSV 交叉校验(差异 >5 条/年告警)。
- **Stooq**:requests 拉 CSV(退市/补缺),标 `source='stooq'`,**不参与 adj_factor 自算**。

### G) capital_flows 表 —— 跨池调拨
新增 `capital_flows` 表;**nav 时间加权(TWR)计算归位领域层**(见 §5 硬约束 2)。`tally capital` 保持 CLI 最小实现,**不在看板做调拨表单**。

## 5. 两条必须落实的新硬约束

> ⚠️ **硬约束 1 —— 跨进程写串行化(T4.3)**
> 双 launchd 开机补触发下,CN 与 US 两个 `tally run` 进程可能**并发写同一 SQLite**,击穿 WAL 的写-写容忍 → `SQLITE_BUSY`。
> **修订:把「单写者」从进程内不变量升级为跨进程全局不变量。最小方案 = 写阶段持 DB 路径文件锁(`flock`),第二进程阻塞等待。写入 T4.3 的 AC。**
> **裁决:launchd + 文件锁 > APScheduler daemon,launchd 结论不变。**

> ⚠️ **硬约束 2 —— nav 时间加权(TWR)归位 `portfolio/ledger.py` 领域层(T4.1)**
> `capital_flows` 跨池调拨的 TWR 计算**放领域层,不放 Repository、不放 service、不内联 runner 或看板**。理由:回测与实盘须同源、只能一份实现。写入 T4.1 的 AC。

**并入 AC 的其他纪律(承接 v2.0 护栏,双市场新增):**

| 纪律 | 落地位置 |
|---|---|
| **零 `if market==` 违例**(AST/grep 可测,broker/profile 豁免) | **M3 硬验收门** |
| **market 维度收口**:所有查询强制带 market 过滤、不提供跨市场混合查询、单市场 run 对另一市场零读写 | Repository / **T4.3 · T4.4 隔离 AC** |
| **深合并列表整体替换语义 + `tally config dump` 工具** | **T0.5.2** |
| **US 全量重拉原子事务** `replace_kline_series`(BEGIN...COMMIT) | Repository / 单写线程 |
| **M2 参数化设计仅实现 CN 分支,M3 不许重写**;建议 **M3 加 diff 审查门**(美股接入 PR 不得改 strategy/ 主逻辑,只增配置与适配器) | M2 → M3 |
| **cache key = `(market, trade_date)`;写后 `st.cache_data.clear()`** | 看板层 |
| **`expired` 展示态由后端判定,前端 disabled 灰显不自算日历** | 前端 / runner |

*v2.0 原有护栏全部延续:分层单向依赖(入口→service→领域→Repository)、三条铁律(SQL 只在 Repository / Streamlit 不直读 SQLite / 看板写走同一写通道)、单写线程 + queue、`ON CONFLICT` 方言纪律、无未来函数、幂等 runner、真值以 DB 为准 / key 用 signal_id。*

## 7. 给项目负责人的决策摘要

1. **v2.2 无需改动技术栈,可直接在 v2.0 基础上继续。** 六大主题选型全部延续并被双市场印证;新增复杂度全在数据接入层与 MarketProfile 抽象,没有任何新常驻组件。这继续是「抗弃坑」的最优解。
2. **两条新硬约束必须落地,否则会出线上事故/数据错误:** ① 双 launchd 补触发下 CN/US 可能并发写库 → 必须加 **DB 文件锁(flock)** 串行化写(T4.3);② 跨池调拨的 **TWR 计算必须归位 `ledger.py` 领域层**(T4.1),保回测实盘同源。
3. **一条可维护性生死线:** MarketProfile 机制 + **「零 `if market==`」作为 M3 硬验收门**,并要求 **M3 美股接入 PR 只增配置与适配器、不改主逻辑**(diff 审查门)。这是单人维护双市场不失控的关键。
4. **脆弱依赖纪律:** yfinance 必须 pin 精确版本;SEC 客户端自研 ~50 行(拒绝 edgartools);数据降级/健康度升格为一等产品特性,**看板与日报须显式展示美股数据健康**(源 / stale / 退市票排除占比)。

### 开放项决议(项目负责人 2026-07-08 已拍板)

- ✅ **美股上线排期:采纳灰度节奏。** `US.enabled` 默认 `false`,直到 A股链路满足「无人工干预连续跑 ≥4 周」(成功标准 2)再开启美股。
- ✅ **通知渠道:改用 Telegram**(推翻上一轮 Slack 裁决)。落地 = `requests` POST 到 Telegram Bot API `sendMessage`,配置 `TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID` 走 `.env`,不引 SDK。**待办:同步修订 `IMPLEMENTATION_SPEC.md`(§0/§8.3/§8.4/§10)将 Slack 全部替换为 Telegram。**
- ☑️ **SEC User-Agent 含个人邮箱(PII):已知悉并登记**「产品化前待处理清单」。一期自用可接受,未来产品化/多用户前须处理。
