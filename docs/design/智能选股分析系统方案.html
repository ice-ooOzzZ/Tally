智能选股分析系统 · 项目方案

每周基本面筛池 · 每日规则化信号 · 回测验证 · 可解释的买卖建议

版本 v1.02026-07-08覆盖 A股 + 美股 定位：自用，预留产品化已通过量化 + 架构双视角评审

1项目概述

1.1 背景与目标

个人投资者无法人工持续跟踪 A股 5000+ 与美股数千只股票。本系统将选股方法论固化为程序：每周按基本面标准维护受监控的股票池（入池/出池），每日对池内股票执行规则化分析，输出「建议买入 / 持仓卖出 / 回避 / 观望」及可解释的触发原因，配套止损价与仓位建议，通过 Web 看板与 IM 推送触达用户。

1.2 设计原则

  1. 可解释：每个信号必须附带人能读懂的原因，拒绝黑盒。
  2. 可验证：策略上线前必须通过历史回测；上线后持续追踪建议的真实表现（成绩单）。
  3. 回测与实盘同源：回测引擎与每日信号共用同一套规则、指标、风控代码，回测只负责喂「截断到 T 日」的数据 —— 这是防未来函数最有效的结构性保证。
  4. 可扩展：规则即插件，新增一条选股规则不改动引擎代码。
  5. 自用优先，预留产品化：单机可跑；模块边界、配置分离、存储访问收口按未来服务化标准设计。

1.3 非目标（一期明确不做）

  • 不做自动下单 / 实盘交易接口（合规与风险原因，仅输出建议）
  • 不做分钟级 / tick 级数据与日内策略
  • 不做多用户账号体系（架构预留）
  • 不做 ML 预测模型（先验证规则策略基线）
  • 不做 A+美混合组合净值与汇率折算（两市场独立子组合、独立基准）

1.4 成功指标

层面指标说明
系统每个交易日最终产出率 ≥ 99%（允许 T+1 补跑）；单日全流程 < 15 分钟个人电脑会睡眠/关机，靠幂等补跑保证最终产出
策略样本外区间年化超额收益 > 0（对各自市场全收益基准）；最大回撤 < 基准 1.2 倍；多头信号胜率 > 45% 且盈亏比 > 1.5训练段表现仅作开发参考，不算达标依据（防重言式）
使用信号原因可读可复核；3 分钟看完日报可解释性验收

2架构审核与总体设计

2.1 对原始四层划分的审核

原始提案：股票层 / 策略层 / 数据层 / 运行层

#审核意见调整
1数据层是股票池层和策略层共同消费的基础设施，应显式置底调整为自底向上分层
2「运行层」混合了调度执行与展示通知两种职责，变更频率与技术栈不同拆分为调度入口与展示层
3缺回测验证 —— 不回测，规则分值与阈值全是主观臆断新增回测与验证模块，一期必做，且设为项目决策门
4缺信号追踪 —— 上线后无法回答「它到底准不准」纳入验证模块：记录每次建议的 T+N 表现
5缺风控辅助 —— 只说买、不说止损和仓位的系统是残缺的纳入策略层输出

2.2 最终架构：核心域模块 + 两个入口 + 横切件

评审修正：回测不是运行时数据流的「一层」，而是横向消费数据/池/策略的另一条执行路径；调度是入口而非中间层。避免「逐层透传」式的过度设计。

入口 A：调度管道幂等 catch-up runner（CLI）· launchd/手动触发 · as-of-date 贯穿
入口 B：Web 看板Streamlit · 信号/池/个股/成绩单/回测实验室
▼ 调用核心域模块 ▼
股票池周度筛选 · 出入池 · 白黑名单 · 行业上限
策略插件规则引擎 · 打分合成 · 止损/仓位
回测事件驱动回放 · 成本模型 · 绩效归因
信号追踪T+1/5/20 表现 · 滚动成绩单
▼ 所有持久化访问必须经 Repository，禁止散落裸 SQL ▼
数据层数据源适配器（AkShare / yfinance / Tushare Pro）· 同步管道 · SQLite(WAL) · 健康检查
横切件：YAML+pydantic 配置中心 · 结构化日志 · 异常告警 · 交易日历

主数据流：数据更新 → 周度筛池 → 日度信号（含风控输出） → 信号追踪 → 报告与推送，由调度管道按交易日历编排；回测走独立入口，消费同一批模块。

3.2 复权处理（评审重点修正）

✗ 已否决的原方案：存前复权价 + 增量更新前复权以当前价为锚，每次除权除息后全部历史价都会变。增量拉取会把新旧两套复权基准拼在一起，K线在除权日附近断裂，均线/MACD 全部失真 —— 静默腐坏，健康检查发现不了。
✓ 最终方案：存不复权原始价 + 复权因子，计算时动态复权增量更新天然安全；信号价与真实成交价可对账；signal_tracking 的 T+N 收益一律基于复权价计算（防除权日假暴跌）。

3.3 同步管道

  • 限频：全局令牌桶（非每线程独立节流）。AkShare 并发 1–2、yfinance 并发 4，请求间隔可配；指数退避重试 3 次。
  • 写入：多线程拉取、单线程批量写入（生产者-消费者队列），SQLite 开 WAL 模式 + busy_timeout。
  • 增量：日K 只拉本地缺失区间（复权因子方案保证正确性）。
  • 失败恢复：失败股票写入 sync_failures 表，下次同步优先补拉；连续 3 天失败升级告警。K线覆盖率不达标（缺最近 3 日）的股票当日跳过信号生成并在报告标注，禁止用残缺数据算指标。
  • 健康检查：每次同步后检查覆盖率、最新日期、除权事件，结果入日志。

3.4 存储设计（SQLite，Repository 收口，预留 PostgreSQL）

索引：kline(market, date)（回测按日取横截面的主路径）、signals(date)、signal_tracking(signal_id)、pool_history(date)。

时区约定：所有 date 字段统一为交易所当地交易日（naive date）；T+N 追踪按各自市场交易日历推算；跨市场对齐仅发生在展示层。

数据质量：停牌日缺K线属正常不补；退市股票标记冻结；财务数据非负校验。

4股票池层

4.1 筛选标准（全部可配，YAML）

A股美股
硬条件 总市值 ≥ 100亿；0 < 动态PE ≤ 40；PB < 8；20日日均成交额 ≥ 1亿；上市满 1 年；非 ST 市值 ≥ $10B；0 < PE ≤ 40；ROE ≥ 12%；日均量 ≥ 100万股
排序满足硬条件后按流动性排序（评审修正：弃用纯市值降序，避免超大盘价值股霸榜）
行业约束单一级行业 ≤ 池子 20%（评审采纳：成本极低，一期必做，防银行地产白酒扎堆）
池大小5030

4.2 出入池机制

  • 周度运行；新满足者入池，记录入池理由。
  • 出池宽限：连续 2 周不满足才出（防临界抖动）；急剧恶化（PE 转负、ST）立即出。
  • 手工白名单（自选股，永不出池）与黑名单（永不入池）。
  • 全部变动写 pool_history；配合 snapshot 周度快照，可复现任意历史日期的筛池结果。

5策略层

5.1 规则引擎（核心扩展点）

@register_rule
class MaCrossRule(Rule):
    name = "ma_cross"
    category = "trend"          # trend / momentum / volume / valuation / risk
    kind = "event"              # event（事件型，带衰减）| state（状态型）
    decay_days = 3              # 事件发生后 N 日内线性衰减计分
    default_weight = 1.0

    def evaluate(self, ctx: StockContext) -> RuleResult:
        """ctx 提供截断到 T 日的K线+指标+基本面+池信息
        返回 RuleResult(score: float, reason: str | None)"""
  • 装饰器注册自动发现：新增规则 = 新增一个类文件，零改动引擎。
  • 权重与启停在 YAML：rules: {ma_cross: {enabled: true, weight: 1.0}}。
  • 事件型 vs 状态型（评审采纳）：金叉/MACD 交叉是事件（带衰减窗口，防「交叉日高分、次日骤降」的信号闪烁）；均线排列、价格分位是状态（持续计分）。

5.2 打分合成（评审重点修正）

⚠ 评审发现：原「简单加总」有两个缺陷① 趋势类规则（金叉、MA20 位置、MACD）高度相关，加总等于同一信息重复计分，看似 5 条原因实际只有 1 条；② 阈值与启用规则数量耦合，多开一条规则阈值语义就变。
  • 两级合成：先在 category 类内聚合（取类内加权均值），再跨类加权合成为归一化总分。
  • 阈值定义在归一化分数上（默认 ±0.6，可配），与规则数量解耦。
  • M3 回测输出单规则归因（每条规则独立的胜率/IC/盈亏比），据此裁剪冗余规则。

5.3 信号语义（评审修正：三类输出，多空不对称）

信号对象含义
建议买入全池股票归一化得分 ≥ 买入阈值，附原因、止损价、仓位建议
持仓卖出仅有历史买入信号的股票得分 ≤ 卖出阈值，或触发止损/出池，可执行
回避无持仓的看空股票信息提示（A股无法做空），不计入可执行信号统计
观望其余入库留档，作为信号追踪的基线对照组

一期内置规则集：均线金叉/死叉与多空排列、MA20 位置、MACD 交叉、RSI 超卖回升/超买、量价配合（3 日方向）、10 日急跌、250 日价格分位。已在原型中验证信号方向正确。

5.4 风控辅助输出（评审修正公式）

  • 止损价：min(入场价×92%, 20日低点, 入场价 − 2×ATR14)，取最宽（配合小仓位的宽止损策略；文档原「最保守」表述有歧义已修正）。
  • 建议仓位：仓位% = min( 2% ÷ 止损距离%, 10% )，其中止损距离% = (入场价−止损价)/入场价。原公式量纲错误（算出的是股数）已修正；10% 上限与回测口径对齐。
  • 风险冗余：A股 T+1 + 跌停制度下止损可能无法按价成交，单笔实际亏损可能超 2% 预算，按 1.5 倍留冗余，报告中明示。

5.5 持仓生命周期状态机（评审新增，回测与实盘共用）

入场(买入信号 + 仓位) ──▶ 持有 ──▶ 退出（四者取先到）:
                                    ① 持仓卖出信号   ② 触发止损价
                                    ③ 出池强制平仓   ④ 最长持有期(默认60交易日)
停牌: 冻结市值不可交易     退市: 按最后交易价清算

6回测与验证层（一期必做，项目决策门）

6.1 历史时点（PIT）数据 —— 回测可信度的前提

✗ 评审否决的原方案：用 pool_history 重建历史池该表上线后才有数据，无法覆盖过去 5 年。若用「今天的池子」回放历史 = 幸存者偏差 + 前视偏差双重污染，回测年化通常虚高 2–4 个百分点，决策门形同虚设。
  • M0 spike（先于一切开发，2–3 天）：验证 Tushare Pro / AkShare 日频估值接口能否拼出 A股历史截面（市值/PE/成交额/ST/上市日期，含已退市股票）；美股验证历史成分股数据集可得性。
  • Tushare Pro（¥200/年）从「备选」提为回测数据一期必选（若 spike 证实免费源拼不齐）。
  • 财务数据一律按 announce_date + 1 交易日 生效（防「3 月用 4 月底才披露的年报」）。
  • 降级方案：若 PIT 截面实在不可得，一期回测收缩为「固定手工池 + 规则验证」，报告显式标注「存在幸存者偏差，收益上偏」，且不得作为决策门依据。

6.2 回测引擎

  • 事件驱动逐日回放；引擎只负责给策略模块喂「截断到 T 日」的数据切片（与实盘同一套代码路径）。
  • T 日收盘出信号，T+1 开盘成交；A股约束：T+1 制度（当日买入不可卖）、停牌禁止成交、开盘即封板（一字板）不可成交。
  • 持仓管理：复用 5.5 状态机；最多 15 只、单票 10% 上限；当日买入信号超出空位时按归一化得分降序、同分按代码排序（确定性，可复现）。

6.3 成本模型

项目A股美股
佣金双边 0.05%（最低5元，保守假设）按 IBKR 阶梯费率
印花税卖出侧，分段取值：2023-08-28 前 0.1%，之后 0.05%—
滑点固定 0.1%（池内已限流动性，可接受）

6.4 绩效与基准

  • 指标：年化收益、最大回撤、夏普、胜率、盈亏比、换手率、逐年分解 + 牛/熊/震荡分段。
  • 基准（评审修正）：一律用全收益指数——沪深300全收益(H00300)、S&P500 TR。价格指数不含分红，会凭空多出约 2%/年的假超额。
  • A股、美股独立子组合、独立基准、独立报告，不做混合净值（消解汇率问题）。

6.5 防过拟合纪律

  • 训练段（2019–2023）/ 样本外段（2024 至今）严格分离；样本外段整个 M3 期间只允许查看 3 次（查看预算制，防反复偷看导致纪律瓦解）。
  • 最优参数需做相邻参数敏感性检查：参数平原可接受，参数尖峰视为过拟合。
  • 每次实验全配置 + git commit hash 入库（backtest_runs），保证半年后可复现。
  • 二期引入 walk-forward 滚动验证。

6.6 信号追踪（实盘影子成绩单）

  • 每日信号生成后，自动记录 T+1 / T+5 / T+20 的复权收益与同期基准收益。
  • 「胜」的主口径 = T+20 跑赢对应基准（评审定案：绝对收益度量的是 beta 不是 alpha）；辅口径展示绝对收益。
  • 买入信号与回避信号分开统计；观望组作基线对照。
  • 看板展示滚动胜率 —— 回答「这系统最近到底准不准」。

7运行层：幂等补跑管道（评审重点修正）

✗ 已否决的原方案：APScheduler 常驻进程 + 固定时刻触发个人电脑会睡眠/合盖/关机，凌晨 05:30 的美股任务大概率漏跑；「99% 按时产出」物理上不可达；补跑还要求全管道支持历史时点重算，被严重低估。
✓ 最终方案：幂等 catch-up runnerrun_pipeline --until today：每次启动扫描「应跑未跑」的交易日清单，逐日补齐。管道全链路支持 as-of-date 参数（同步→筛池→信号→追踪→报告），这是 M1/M2 的硬性设计约束。
  • 触发：macOS launchd（支持错过后唤醒补触发，优于 cron）+ 允许手动触发；管道本身线性依赖，顺序脚本即可，无需任务编排框架。
  • 节奏：每交易日 A股收盘后跑 A股链路；美股链路北京时间次日早晨跑，日报合并展示（各自记各自市场的交易日）。每周一筛池。
  • 失败处理：单任务重试 3 次（指数退避）→ 最终失败推送告警；单只股票失败不阻塞整体，报告标注。
  • 幂等性：同一 as-of-date 重复运行结果一致（靠主键 upsert + 确定性排序保证）。

11测试策略（评审新增章节）

本系统最大质量风险不在规则逻辑，而在数据正确性与回放正确性 —— 普通单测覆盖不到，需专门手段：

对象测试手段
数据源适配器录制回放：录真实 API 响应做 fixture，CI 不打真实接口
技术指标与 TA-Lib / 手算已知值对照
回测引擎黄金用例：3 只股票 × 20 天，人肉核对每笔成交与净值曲线
防未来函数同一规则分别喂「截断至 T 日」与「全量」数据，T 日信号必须逐字节一致
复权正确性除权日跨越的 T+N 收益计算回归测试
幂等性同一 as-of-date 重复跑管道，产物必须一致
规则单元每条规则配已知形态K线（上涨/下跌/横盘/急跌）断言信号方向

12里程碑（按业余小时估算，评审修正）

阶段内容估算验收标准
M0 · Spike回测 PIT 数据可行性验证（Tushare/AkShare 历史截面、美股历史成分）15–20h能否重建 2020 年任一周的筛池输入？决定回测方案走主路径还是降级路径

M3 · 回测
决策门		回放引擎、broker 约束、成本模型、归因、样本外验证70–90h5 年回测跑通；防未来函数与黄金用例测试通过；样本外基线不达标则调整规则集，不硬上

M5 · 试运行信号追踪积累、按回测归因调整权重持续 8–12 周过程性验收：管道稳定、样本持续积累；胜率评估推迟到累计 ≥ 100 个信号（评审修正：4 周几十个信号的胜率置信区间 ±10–15pp，统计上不可判定）

合计一次性开发约 215–290 业余小时；按每周 10 小时投入约 5–7 个月。评审共识：数据层的长尾（接口变更、反爬、边界情况）占实际时间 50% 以上，宁可估宽。
